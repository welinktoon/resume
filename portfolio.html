01 Цель продукта

Создать внутреннюю рабочую панель для продуктовой команды, которая собирает задачи, встречи, источники данных и статусы подключений в одном интерфейсе.

Главная задача: сократить ручной обход сервисов и быстрее подсвечивать риски, которые влияют на delivery.

02 Границы MVP

Входит в MVP
  • единая сводка по задачам, встречам и источникам;
  • фильтры по просрочкам, критичности и отсутствию движения;
  • статусы подключений и понятные ошибки синхронизации;
  • переходы из панели в исходные объекты: задачу, встречу, таблицу.
Не входит в MVP
  • редактирование задач и встреч внутри панели;
  • автоматическая рассылка статусов во внешние чаты;
  • сложная ролевая модель с правами на каждый виджет;
  • историческая аналитика за несколько кварталов.

03 Пользователи

Системный аналитик
  • видит задачи, блокеры, статусы и комментарии;
  • готовит вопросы к бизнесу и разработке;
  • проверяет готовность сценариев к приемке.
Продакт / проектный / delivery-менеджер
  • контролирует риски, сроки и загрузку недели;
  • быстро понимает, где нужен созвон или эскалация;
  • видит состояние подключений без ручного запроса статусов.

04 Источники данных

Источник Данные Ключевые проверки
Yandex Tracker задачи, статусы, приоритеты, сроки, комментарии просрочка, критичность, отсутствие движения, наличие комментариев
Google Calendar встречи, плотность недели, ближайшие события нагрузка по дням, ближайшая встреча, конфликтные окна

05 Ключевые сущности и поля

Сущность Обязательные поля Зачем нужны
Task id, title, status, priority, assignee, dueDate, updatedAt, sourceUrl построение доски, фильтров, просрочек и ссылок в первоисточник
Meeting id, title, startsAt, endsAt, organizer, attendeesCount, sourceUrl недельная сетка, ближайшие события и оценка нагрузки
Connection source, status, lastSyncAt, errorCode, errorText, owner контроль интеграций и понятное объяснение, почему данные не обновились
RiskSignal type, severity, reason, relatedEntityId, actionHint единая подсветка блокеров, просрочек и задач без движения

06 API-методы MVP

Метод Endpoint Назначение Параметры / тело Ответ
GET /api/tasks получить задачи для доски, фильтров и счетчиков status, assignee, riskOnly, updatedBefore Task[], meta.syncedAt, meta.total
GET /api/calendar/events получить встречи недели и ближайшее событие dateFrom, dateTo, owner Meeting[], busyDays[], nextEvent
GET /api/connections показать состояние подключений и ошибки источников source, status Connection[], lastSyncAt, errorText
POST /api/sync/run запустить ручную синхронизацию источника { source, force } syncId, status, startedAt
GET /api/risks/signals получить активные сигналы риска severity, type, entityType RiskSignal[], actionHint
PATCH /api/signals/{id} обновить статус сигнала { status, comment } RiskSignal

Общий формат ошибки: errorCode, errorText, source, retryAvailable. Если источник недоступен, интерфейс показывает кэш и время последней успешной синхронизации.

07 Функциональные требования

ID Требование Критерий приемки
FR-01 Показывать сводку по задачам, рискам, встречам и источникам. Пользователь видит ключевые счетчики без перехода в другие разделы.
FR-02 Разделять задачи по статусам: бэклог, ожидание, в работе, тестирование, закрыто. Каждая задача попадает в корректную колонку по статусу из трекера.
FR-03 Подсвечивать просроченные, критичные и зависшие задачи. Задачи с рисками имеют визуальный сигнал и доступны через фильтр.
FR-04 Показывать недельный календарь встреч и ближайшее событие. Пользователь видит количество встреч, плотный день и расписание недели.
FR-05 Показывать статусы подключений к источникам. Для каждого источника виден статус: подключено, ошибка, требуется настройка.

08 Пользовательские истории

Роль История Ценность
Аналитик Я хочу быстро видеть задачи без движения и открытые вопросы, чтобы подготовиться к статусу. меньше ручной сверки перед созвоном
Delivery-менеджер Я хочу видеть риски по срокам и ошибки источников на одном экране, чтобы вовремя эскалировать. раньше обнаруживаются блокеры
Продакт Я хочу видеть встречи недели рядом с задачами, чтобы понимать нагрузку и фокус команды. проще планировать обсуждения и приемку

09 Основные сценарии

Ежедневный обзор
  • пользователь открывает дашборд;
  • видит риски, встречи, просрочки и сбои источников;
  • переходит только в те зоны, где нужно действие.
Подготовка к статусу
  • пользователь фильтрует задачи без движения;
  • сверяет встречи недели;
  • формирует список вопросов и следующих действий.

10 Правила сигналов

Сигнал Условие Действие в интерфейсе
Просрочено dueDate меньше текущей даты, статус не закрывающий красная метка, попадание в фильтр рисков
Нет движения updatedAt старше 7 дней и задача не закрыта желтая метка и подсказка проверить актуальность
Ошибка источника последняя синхронизация завершилась ошибкой или истек токен статус "требует внимания" и короткий текст причины
Плотный день 4+ встречи в рабочий день подсветка дня в календаре и счетчик нагрузки

11 Нефункциональные требования

  • Обновление данных не должно блокировать просмотр уже загруженной информации.
  • Ошибки источников должны быть видны пользователю понятным текстом.
  • Интерфейс должен работать на десктопе и планшете без горизонтального скролла страницы.
  • Чувствительные данные в публичных скриншотах должны быть скрыты или заменены.

12 Статусы и сигналы

просрочено критично нет движения есть комментарии подключено ошибка синхронизации

13 Критерии приемки

  • Панель отображает задачи, встречи и источники на одной странице.
  • Риски по задачам доступны без ручной проверки трекера.
  • Состояние подключений видно до запуска ручной синхронизации.
  • При недоступности источника пользователь получает понятный статус и текст ошибки.
Диаграммы и API

Поток данных и сигналы

Схемы показывают, как внешние источники попадают в рабочую панель, нормализуются и превращаются в сигналы для команды.

GET /api/tasks

Список задач для доски, фильтров, счетчиков и сигналов риска.

GET /api/calendar/events

Встречи недели, ближайшее событие и плотность рабочего дня.

GET /api/connections

Состояние источников, время синхронизации и понятный текст ошибки.

POST /api/sync/run

Ручной запуск синхронизации источника с возвратом syncId и статуса.

GET /api/risks/signals

Активные сигналы по задачам, встречам и источникам.

PATCH /api/signals/{id}

Обновление статуса сигнала: взято в работу, закрыто, нужен комментарий.
